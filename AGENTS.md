# This is NOT the Next.js you know

This version has breaking changes — APIs, conventions, and file structure may all differ from your training data. Read the relevant guide in `node_modules/next/dist/docs/` before writing any code. Heed deprecation notices.

Routing and auth are locale-first: user-facing pages live under `src/app/[lang]/`, `src/app/page.tsx` redirects to `defaultLocale`, and request guarding/locale redirects are implemented in `src/proxy.ts`.
When changing authentication redirects or locale path handling, keep `src/proxy.ts`, `src/lib/auth/redirect.ts`, and `src/components/language-switcher.tsx` in sync.
Route-level data mutations and reads are implemented as server actions in colocated `actions.ts` files (for example `src/app/[lang]/actions.ts`, `src/app/[lang]/login/actions.ts`, `src/app/[lang]/get-points/actions.ts`) and consumed by client forms via `useActionState`.
Local dev workflow is `pnpm db:up` -> `pnpm db:check` -> `pnpm dev` (see `README.md`).

This is a pnpm project, so make sure to use `pnpm` instead of `npm`/`npx` for commands.

## Component library

This project uses shadcn/ui as a component library.
All UI components must be built using this library, and no custom CSS or HTML should be used for UI elements.
Use the shadcn MCP server to find the right components and their usage.
Prefer importing primitives from `src/components/ui/*` wrappers (for example `button.tsx`, `card.tsx`, `table.tsx`, `dropdown-menu.tsx`) instead of directly using third-party base primitives in feature components.

## Database

This project uses Drizzle ORM for database access, with a Postgres database.
DO NOT change any files inside the `drizzle/` folder.
These files are autogenerated via `drizzle-kit generate`.
Apply them by running `drizzle-kit push`.

## i18n

This repository uses internationalization or i18n via Next.js.
No user-facing text must be added directly to *.tsx files, but instead added to src/i18n/dictionaries/de.ts and src/i18n/dictionaries/en.ts and then be referenced in the *.tsx files.
Keep dictionary keys aligned between `src/i18n/dictionaries/en.ts` and `src/i18n/dictionaries/de.ts`, including metadata keys used by `generateMetadata` in `src/app/[lang]/layout.tsx`.
Validate locale params with `hasLocale` from `src/i18n/config.ts` and call `notFound()` for invalid locales before rendering page content.
Load dictionaries through `getDictionary` from `src/i18n/get-dictionary.ts` in server components, then pass the required dictionary slice down as typed props.

## Dark mode

This project supports both dark and light mode, via the NextThemesProvider.
All changes must make sure to not break either the light or dark modes.
Prefer theme tokens such as `bg-background`, `text-foreground`, and `border-border`; if a fixed light color is used, provide an explicit `dark:` counterpart.

## Responsive design

This project is meant to be used in web browsers on both PCs as well smartphones.
All changes to the UI must make sure to comply to both an upright smartphone layout as well as a widescreen PC layout. 
Follow existing layout patterns like `px-4 sm:px-6`, `max-w-*` wrappers, and mobile-safe full-width forms (`w-full`, optional `sm:w-auto`).
